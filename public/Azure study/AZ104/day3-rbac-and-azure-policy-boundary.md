# はじめに

AZ-104取得を目標に学習中。

Day3は **RBAC と Azure Policy の役割と限界を整理すること** に集中した。

単なる機能理解ではなく、

- それぞれが何を制御しているのか
- どこまで守れて、どこから守れないのか
- ガバナンスの限界はどこにあるのか

を言語化することを目的にした。

---

# 1. なぜガバナンスが必要なのか？

ガバナンスは「責任者を決める仕組み」ではない。

本質は

> リスクを管理可能な状態に保つこと

事故をゼロにするのではなく、

- 事故を減らす
- 影響範囲を小さくする
- 早く検知する
- 復旧しやすくする

ための仕組みである。

---

# 2. RBACとは何を制御しているのか？

RBAC（Role-Based Access Control）は

> 「誰が」「どこに対して」「何の操作をできるか」を制御する仕組み

つまり **操作権限の制御**。

- スコープ（管理グループ / サブスクリプション / リソースグループ / リソース）
- ロール
- 割り当て対象（ユーザー / グループ / サービスプリンシパル）

で構成される。

## 気づき

RBACは「許可範囲」を制御するが、

- 操作が正しいかどうか
- 意図が妥当かどうか

までは判断しない。

つまり

> 許可された誤りは防げない

---

# 3. Azure Policyは何を制御しているのか？

Azure Policyは

> リソースの構成状態が許容されるかどうかを評価・制限する仕組み

代表的な効果：

- Deny（作成・更新を拒否）
- Audit（違反を記録）
- Modify / DeployIfNotExists など

RBACが「操作」を見るのに対して、  
Policyは「状態」を見る。

## 気づき

Policyは構成の逸脱を防げるが、

- 誤削除
- データ操作
- アプリケーションのバグ

までは防げない。

---

# 4. RBAC × Policy は冗長か？

結論：

> 冗長ではなく補完関係

- RBAC → 誰が触れるか
- Policy → 触った結果が許容範囲か

どちらか単体では統制は不十分。

---

# 5. 管理プレーンとデータプレーン

今回の最大の学び。

RBAC / Policy が主に制御するのは

> 管理プレーン（Azure Resource Manager経由の操作）

例：
- リソース作成
- 削除
- 設定変更

一方、

> データプレーンはリソース内部の操作

例：
- DBのクエリ実行
- VM内でのコマンド実行
- Storageへのデータ書き込み

ここはRBACやPolicyだけでは守れないケースがある。

## 重要な気づき

管理プレーンを完璧に統制しても、

- データ消失
- コスト爆発
- パフォーマンス劣化

は起き得る。

なぜなら、

> 設計の質・運用・想定外の振る舞いまでは制御できないから

---

# 6. ガバナンスの限界

RBAC完璧  
Policy完璧  
Lock完璧  

それでも事故は起きる。

理由は、

- 想定外の設計ミス
- データプレーンでの操作
- 運用の逸脱
- 時間とともに変化する権限や組織

つまり

> ガバナンスは事故をゼロにする仕組みではない  
> リスクを制御可能な範囲に抑える仕組みである

---

# 7. 今日の到達点

理解できたこと：

- RBACとPolicyの明確な役割分担
- 管理プレーンとデータプレーンの違い
- 統制の限界の存在

まだ弱いこと：

- 事故半径の具体化
- 実務的なシナリオ設計
- 監視・運用レイヤーの統合

---

# まとめ

RBACやPolicyは強力だが万能ではない。

アーキテクト視点で見ると、

- 権限設計
- 構成制約
- 監視
- 運用
- 復旧戦略

すべてが揃って初めて「統制」になる。

Day3は  
**制御レイヤーの境界を理解する日**だった。

次はこの理解を、実装や監視設計に落とし込んでいく。
