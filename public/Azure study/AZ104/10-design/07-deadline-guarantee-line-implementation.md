# はじめに

本稿では、前稿で定義した「提出可能状態の保証」を
運用可能な実装パターンに落とし込む。

---

# 1. 先に固定する4項目

- 締切（Deadline）
- 回復時間（Recovery Time）
- 判定時刻（Gate Time）
- 最終手段（Fallback）

この4項目が曖昧なままでは、監視を増やしても締切は守れない。

---

# 2. 判定は二択化する

運用判断を増やすほど遅延するため、判定は次の二択に寄せる。

- 正常：通常ルート継続
- 異常：代替ルートへ即時切替

「様子見」を標準動作にしない。

---

# 3. 最小実装テンプレート

```text
T_deadline: 提出締切
T_recover : 回復に必要な時間
T_gate    : T_deadline - T_recover

if now >= T_gate and 提出可能条件を満たさない:
    代替ルートへ切替
else:
    通常ルート継続
```

---

# 4. 提出可能条件を機械判定にする

- 入力完了：必要データが揃っている
- 生成完了：成果物が生成済み
- 配送可能：提出経路が利用可能
- 検証完了：最低限の整合性チェック通過

いずれかが欠ければ提出可能状態ではない。

---

# 5. 監視指標の中心を変える

監視は件数中心ではなく、締切余裕中心で設計する。

- 残り余裕（分）
- 回復見積もり時間（分）
- 判定時刻との差分（分）

これでアラートは「異常通知」から
「締切保証の毀損通知」に変わる。

---

# 6. 結論

> 締切制約システムは、時刻条件と切替条件を先に固定し、判定を二択化して初めて運用可能な保証ラインになる。
