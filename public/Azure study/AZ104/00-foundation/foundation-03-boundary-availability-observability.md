# はじめに

この回では、Azure機能の暗記ではなく
「なぜその境界にするのか」を基準に整理した。

扱う論点は次の3つ。

- ネットワーク境界（VNet / Subnet / NSG）
- 可用性設計（RTO / RPO と冗長化）
- 観測性（メトリクス / ログ / 相関ID）

---

# 1. ネットワーク境界

## VNet

VNetはIP空間の分離だけでなく、
障害影響範囲を制御する境界。

## Subnet

Subnetは責務と制御を分離する単位。

- セキュリティルールの適用単位
- 変更頻度の違いを吸収する単位
- スケール単位の分離

---

# 2. NSG設計で押さえる点

NSGは「最初に一致したルールで評価終了」する。

そのため、ルール定義は次を明確にする。

- どこから（Source）
- どこへ（Destination）
- 何を（Protocol/Port）

ポートだけの許可ではなく、通信元境界まで絞る。

---

# 3. 可用性設計

可用性設計は、先にRTO/RPOを決める。

- RTO：復旧までに許容される時間
- RPO：損失してよいデータ量

この2つを決めてから、
可用性セット・ゾーン・リージョン冗長を選択する。

---

# 4. 観測性

運用で必要なのは「検知」だけでなく「追跡可能性」。

- メトリクス：異常の兆候を見る
- ログ：詳細原因を掘る
- 相関ID：境界を跨いだ追跡を可能にする

相関IDがないとMTTRは大きく悪化する。

---

# まとめ

- 境界はネットワーク分離のためだけでなく、影響範囲制御のために設計する
- 可用性はSLA参照より先にRTO/RPOで決める
- 観測性は相関IDを含めて初めて機能する
